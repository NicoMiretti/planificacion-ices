# Protocolo de Entrevistas

## Antes de la entrevista
- [ ] Definir objetivo concreto (ej: "entender el flujo actual de aprobación").
- [ ] Enviar agenda al entrevistado 24-48 hs antes (temas, duración, no requiere preparación técnica).
- [ ] Pedir consentimiento para grabar (audio o notas).
- [ ] Tener a mano: glosario preliminar, lista de preguntas, hoja para diagramar.
- [ ] Preparar pregunta de cierre: *"¿Hay algo importante que no te pregunté?"*.

## Durante la entrevista
- **Duración recomendada:** 45–60 min. Cortar antes si decae la atención.
- **Estructura:**
  1. Presentación (2 min): quién sos, objetivo, cómo se usa la info.
  2. Contexto del entrevistado (5 min): rol, antigüedad, día típico.
  3. Proceso actual "AS-IS" (15–20 min): cómo se hace hoy, sin hablar de software.
  4. Dolores y deseos (10 min): qué falla, qué le gustaría.
  5. Proceso futuro "TO-BE" (10 min): cómo se imagina con el sistema.
  6. Cierre (5 min): próximos pasos, validación, agradecimiento.
- **Técnicas:**
  - Preguntas **abiertas** primero ("contame cómo…"), cerradas para confirmar.
  - **5 porqués** cuando aparece un problema.
  - **No proponer soluciones** durante el relevamiento.
  - Repetir lo entendido con tus palabras ("a ver si entendí…").
  - Anotar verbatim frases clave (sirven luego para casos de uso).
  - Si menciona un documento/formulario, **pedir copia**.

## Después de la entrevista
- [ ] Pasar notas a limpio en las primeras 24 hs.
- [ ] Completar **acta** en `docs/02-relevamiento/actas/`.
- [ ] Actualizar glosario, stakeholders y reglas de negocio detectadas.
- [ ] Enviar acta al entrevistado para validación (responder con "OK" basta).
- [ ] Listar dudas pendientes para próxima ronda.

## Errores frecuentes a evitar
- Hacer preguntas que sugieren la respuesta.
- Mezclar AS-IS con TO-BE → confunde al usuario.
- Hablar más que el entrevistado.
- Asumir que un término tiene el mismo significado para todos (validar contra glosario).
- Salir sin próximos pasos definidos.
